The Problem: AI Without Boundaries

You're building software with AI assistance. You ask it to help design a feature. The AI is intelligent, helpful, and... completely unaware of your constraints.

The "Martians" Analogy:

Imagine Martians land on Earth. They're intelligent - they can build anything. But they don't know Earth's laws, customs, or constraints. So they confidently build things that seem right to them but violate rules they never knew existed.

AI systems are Martians. They can build anything, but without explicit context about YOUR world, they'll invent solutions that seem correct but code you into corners.

The Result: Architectural Drift

  • AI invents features outside your scope ("Let me add team collaboration!" when you wanted single-user)
  • AI uses the wrong patterns (FHIR everywhere when you decided it's only for EHR integration)
  • AI ignores constraints it never knew existed (security requirements, data handling rules)
  • Every session starts fresh - no memory of previous architectural decisions

The Real Cost

The AI produces technically correct but architecturally wrong code. You discover this during review. You refactor. Repeat.

What You Asked

"Create a Task model for our task management system"

What You Got

A Task model with team collaboration, assignments, attachments, notifications - features you explicitly scoped out.

The AI isn't wrong. It's uninformed.

It doesn't know your boundaries because you never told it. Not once - you need to tell it every session, every time.

Scene 2 of 6

What is SCS?

Structured Context Specification (SCS) is an open standard for defining the operating environment that AI systems work within. It gives AI the "world" it needs to make decisions that align with YOUR constraints.

The Core Idea:

Instead of telling AI your constraints every session, you define them once in a structured, versioned format. The AI loads this context and operates within those boundaries.

Core Concepts: SCDs and Bundles

SCD: Structured Context Document

An SCD is a single, atomic definition of one essential thing: a rule, a boundary, a constraint, or a domain concept.

id: scd:architecture:boundary-system-scope version: "0.1.0" domain: architecture type: boundary title: "System Scope Boundary" boundary: inside: - "Task creation and management" - "User authentication" - "Task completion tracking" outside: - "Team collaboration features" # AI now knows: DON'T build this - "Task assignment to others" # AI now knows: DON'T build this - "Calendar integration" # AI now knows: DON'T build this - "Notifications or reminders" # AI now knows: DON'T build this - "File attachments" # AI now knows: DON'T build this

Key properties of a good SCD:

  • Atomic: Defines exactly one concept
  • Versioned: Has a semantic version (0.1.0)
  • Traceable: Includes provenance (who created it, why)
  • Machine-readable: YAML structure AI can parse

Bundle: Versioned Collection

A Bundle assembles multiple SCDs into a complete operating environment. Think of it as a "release" of your context.

Bundle Types

Type Purpose Contains
Project Top-level orchestrator Imports only (no SCDs directly)
Domain Single concern area SCDs only (no imports)
Standards Compliance requirements External standards (HIPAA, SOC2)
Meta SCS foundation vocabulary Core SCS definitions
The Key Insight: Project bundles import domain bundles. Domain bundles contain SCDs. This hierarchy lets you compose complete environments from reusable pieces.

The Bundle Lifecycle

SCS follows a context-first approach: define and lock your context before building. Bundles progress through a clear lifecycle.

1
DRAFT
Create & iterate
2
Validate
Check correctness
3
Version
Assign semver
4
Lock
Immutable contract

Phase 1: DRAFT

Create SCDs and bundles. Iterate freely. Work with AI to structure your knowledge.

version: "DRAFT" # Can be modified

Phase 2: Validate

Run validation to ensure structure is correct, references are valid, and domains are assigned.

$ scs validate ./bundles/my-project.yaml ‚úì All SCDs have unique IDs ‚úì All SCDs have versions ‚úì All domains are valid (11 canonical) ‚úì All bundle references resolve ‚úì Bundle is ready for versioning

Phase 3: Version

Assign a semantic version. This signals the bundle is ready for use.

version: "1.0.0" # Semantic versioning

Phase 4: Lock (Immutable)

Once versioned and committed, the bundle is immutable. It becomes a contract. Changes require a new version.

Why Immutability Matters:
  • You can always trace what context the AI had at any point in time
  • Reproducible behavior - same bundle = same AI boundaries
  • Audit trail - prove what rules the AI was following
  • Rollback - if v1.1 has issues, revert to v1.0

Context-First Development

Define your boundaries. Version them. Then build. Changes to context go through the same lifecycle - new version, validate, lock.

Before & After: Same AI, Different Behavior

Let's see the difference in action. Same AI model, same prompt - the only difference is whether SCS bundles are loaded.

The Task

Prompt: "Create a Task model for our task management system"

WITHOUT SCS Bundles

AI invents features based on training data:

class Task:
    def __init__(self, title, description,
                 assignee,       # NOT in scope!
                 team_id,        # NOT in scope!
                 priority, tags):
        self.title = title
        self.description = description
        self.assignee = assignee
        self.team_id = team_id
        self.priority = priority
        self.tags = tags
        self.attachments = []  # NOT in scope!

WITH SCS Bundles

AI respects your boundaries:

class Task:
    """
    Task owned by a User.
    Per system scope boundary, this is
    single-user only with no team features.
    """
    def __init__(self, title, description,
                 owner_id):
        self.title = title
        self.description = description
        self.owner_id = owner_id
        self.completed = False

What Changed?

Aspect Without SCS With SCS
Respects scope boundary No (invents features) Yes
Team features Added (not requested) Excluded (per boundary)
Attachments Added (not requested) Excluded (per boundary)
References constraints No Yes (docstring)
Code alignment Training data YOUR architecture
Rework needed Yes No

The AI didn't get smarter. It got informed.

Same model, same capabilities. But with SCS, it operates within YOUR world instead of inventing one from training data.

Scene 6 of 6

Key Takeaways

1. AI Systems Need Boundaries
Without explicit context, AI improvises. It's not wrong - it's uninformed. SCS provides the boundaries AI needs to align with YOUR architecture.
2. Define Once, Use Everywhere
SCDs and bundles are reusable. Define your architecture decisions once. Load them into any AI session. Get consistent behavior across your entire team.
3. Versioned Contracts
Bundles are immutable once versioned. You always know what context the AI had. Reproducible behavior. Auditable decisions.
4. Context-First Development
Define the environment before building. Human + AI collaboration: you provide domain expertise, AI helps structure it into SCDs.
5. Open Standard
SCS is an open specification. No vendor lock-in. Works with any AI system that can consume structured context. Build on a foundation you control.

The Bottom Line

Without Structured Context: AI that invents features, ignores constraints, and produces technically correct but architecturally wrong code.


With Structured Context: AI that operates within YOUR boundaries, respects YOUR decisions, and produces code that aligns with YOUR architecture from day one.
